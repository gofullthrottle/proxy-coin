01 Steel Thread PoC Week 1–2
Step 1
Protocol Buffers
Define wire protocol consumed by both Go backend and Kotlin Android app.
98%
Implementation
Define 5 message types in protocol/proto/proxy.proto: RegisterRequest, RegisterResponse, Heartbeat, ProxyRequest, ProxyResponse. Wrap in an envelope Message type. Generate code for Go (protoc --go_out) and Kotlin (protobuf-kotlin-lite). Estimated 2-4 hours.
Why 98%
Protobuf is mature tooling used by Google, gRPC, and thousands of production systems. The only risk is schema design mistakes, which are cheap to fix early. Zero technical uncertainty.
If It Fails
Protobuf can't fail in a meaningful way. If you hit tooling issues with Kotlin Lite, fall back to JSON temporarily and migrate later. This is a non-blocking step.
Step 2
Go WebSocket Server
Minimal orchestrator: accept node connections, route test proxy requests.
95%
Implementation
Single main.go (~400 lines). Two endpoints: ws://host:8080/ws for node WebSocket connections, http://host:8081/test?url=X for test proxy requests. Uses gorilla/websocket (20K+ GitHub stars) and sync.Map for in-memory node registry. No database needed for PoC.
Why 95%
Go WebSocket servers are well-understood. Discord, Slack, and Coinbase all run Go WebSocket infrastructure. The gorilla/websocket library handles compression, ping/pong, and graceful shutdown. The 5% risk is subtle concurrency bugs in request routing, which testing will catch.
Deploy Options
Run locally (go run), on homelab Docker Swarm, or on a $5/mo VPS. Go compiles to a single ~10MB static binary with zero dependencies. Can test with websocat CLI before the Android app exists.
Step 3
Android Foreground Service
Minimal app: WebSocket connection, HTTP proxy execution, persistent notification.
80%
Implementation
Two files that matter: ProxyForegroundService.kt (WebSocket + proxy engine) and MainActivity.kt (ON/OFF toggle). The service uses OkHttp WebSocket client, executes proxy requests with OkHttp HTTP client, and shows a persistent notification. Must request REQUEST_IGNORE_BATTERY_OPTIMIZATIONS, hold PARTIAL_WAKE_LOCK and WifiManager.WifiLock.
Why 80% (not higher)
The 20% risk is Android OEM background restrictions. Samsung (One UI), Xiaomi (MIUI), Huawei (EMUI), and Oppo (ColorOS) all have aggressive battery optimization that kills background services regardless of WakeLocks and START_STICKY. Stock Android (Pixel) works reliably. The mitigation is a device compatibility list and manufacturer-specific workarounds (dontkillmyapp.com documents these).
Critical Gotchas
1. onTaskRemoved() — must restart service when user swipes app away
2. FOREGROUND_SERVICE_DATA_SYNC type required (Android 14+)
3. Battery optimization exemption dialog — user must explicitly allow
4. Samsung "Sleeping apps" — must guide user to exclude the app
5. Notification channel importance — must be at least DEFAULT
Step 4
Round-Trip Validation
curl → backend → WebSocket → phone → HTTP → response. Phone's residential IP visible.
90%
The Test
curl "http://orchestrator:8081/test?url=https://httpbin.org/ip"

Expected: {"origin": "73.xxx.xxx.xxx"} — the phone's residential IP, NOT the server's IP.

This single test validates the entire business model. If the response shows a residential IP that differs from the server's IP, proxy-through-mobile works.
Why 90%
Conditional on steps 1-3 working, this is almost certain to succeed. The 10% risk is edge cases: some corporate WiFi networks block outbound WebSocket, some ISPs use CGNAT that makes the IP appear as datacenter, or DNS resolution on the phone fails for the target URL.
Continue to Phase 2
  • Proxy success rate > 85%
  • Battery drain < 5%/hour
  • WebSocket stable 24h (< 5 reconnects)
  • Latency overhead < 300ms
Reassess / Pivot
  • Success rate < 50%
  • Battery drain > 10%/hour
  • Constant disconnects (> 20/day)
  • Pivot: compute-only or desktop-first
|| Legal Opinion (Parallel Track) Week 2 → Week 10
Parallel
Token Classification Opinion
Crypto law firm assesses PRXY under Howey test, MiCA, and FinCEN frameworks.
70%
Why Parallel
Lawyers take 4-8 weeks from engagement. Token development starts at week 12. If you start legal at week 2, opinion arrives at week 6-10 — perfectly timed. Starting later creates a 4-8 week delay in the critical path.
Why 70% for Utility Classification
Strong utility argument: PRXY has immediate use (earn it by sharing bandwidth, spend it for proxy access, stake it for multipliers). No "common enterprise" — users earn through their own device resources. But the SEC's 2023-2024 enforcement actions expanded what they consider securities. Post-Ripple, the landscape is more favorable for utility tokens, but not certain. Similar projects (Helium/HNT, Filecoin/FIL) obtained utility classification.
If Classified as Security
Fallback: points-based system (Grass model). Users earn points, which convert to tokens later once regulatory path is clear. This is a 3-6 month delay, not project death. Grass ran points-only for 18 months before launching GRASS token.
Cost
$15,000-$40,000 for comprehensive opinion. Firms: Anderson Kill, Cooley, Fenwick & West, or crypto-native boutiques. Cheaper option: use a crypto-focused solo practitioner for initial assessment ($3-5K), then full firm opinion if initial assessment is positive.
02 Demand Validation Week 6–8
Business
Manual Proxy Sales
10-20 test devices. Offer proxy access at 50% of Bright Data's rate to validate demand.
55%
Why This Matters
Technology without customers = expensive hobby. The residential proxy market is proven ($1.2B), but breaking in requires either significantly lower price OR features incumbents lack. Our price advantage (token-based compensation is cheaper than cash payouts) gives us 30-50% price advantage vs Bright Data ($5-15/GB → our $3-5/GB).
Why 55%
Finding 3+ paying customers in 8 weeks is achievable but uncertain. Positive factors: clear price advantage, proven market need. Negative factors: enterprise sales cycles are long (4-12 weeks), trust in new provider is low, small network has limited geo coverage. Most likely early adopters: small web scraping startups and indie developers who are price-sensitive and willing to try new tools.
Outreach Channels
1. r/webscraping — 50K members, active community
2. Web scraping Discord servers (ScrapingBee, Crawlee communities)
3. Direct outreach to companies on G2's "web scraping" category
4. Product Hunt launch of free tier API
5. Dev.to and Hacker News "Show HN" posts
Revenue
Unit Economics Validation
Prove that revenue per GB exceeds cost per GB at small scale.
65%
Target Economics
Revenue per GB (customer pays): $4.00
Node operator reward: $2.80 (70%)
Infrastructure: $0.40 (10%)
Margin: $0.80 (20%)

Break-even: ~19,000 GB/month at these margins.
At 100 nodes × 5 GB/day: 15,000 GB/month → $60K revenue potential.
Why 65%
Market rate validation is strong ($5-15/GB from competitors). The risk is that at small scale, infrastructure cost per GB is higher (fixed costs amortized over fewer GBs) and quality issues (disconnects, high latency) may reduce the premium customers will pay.
Proceed to Token Development
  • 3+ paying customers or LOIs
  • Legal opinion: utility token viable
  • 100+ nodes stable for 1 week
  • Revenue > $500/month
Pivot Options
  • No customers after 12 weeks
  • Token classified as security → use points
  • Network quality too unreliable
  • Pivot: compute-only, desktop-first, or white-label
03 Points System & Dashboard Week 9–12
Android
Dashboard UI
Jetpack Compose: earnings, bandwidth stats, connection toggle, settings.
90%
Why 90%
UI development is the lowest-risk part of the project. Jetpack Compose is mature, well-documented, and has strong tooling. The UI is straightforward: status card, earnings chart, settings toggles. No novel interaction patterns needed.
Backend
Points Ledger & Anti-Fraud v1
PostgreSQL earnings tracking, server-side metering, basic behavioral fraud detection.
85%
Why 85%
Database-backed metering and earnings is standard engineering. The 15% risk is in anti-fraud: behavioral analysis requires tuning, and false positive rates may be too high initially. Iterative improvement expected.
Strategy
Open Source Proxy Engine
Publish the Android proxy engine (not backend) on GitHub for trust and community.
75%
Why 75% for Adoption Impact
Crypto-native users strongly prefer open-source. Signal vs WhatsApp is the analogue. However, open-source alone doesn't drive adoption — it's a trust multiplier on top of a working product. Competitors will copy the code (80% likely) but that's fine — the proxy engine is commodity, the network is the moat.
04 Token Launch Week 12–16
Blockchain
Smart Contract Deployment
ProxyCoinToken, RewardDistributor, Staking, Vesting — deploy to Base L2.
88%
Why 88%
ERC-20 tokens and Merkle distributors are well-established patterns with battle-tested OpenZeppelin implementations. The 12% risk is in the security audit: auditors may find issues requiring contract redesign, which adds 2-4 weeks. Foundry's fuzz testing reduces but doesn't eliminate this risk.
Android
Wallet Integration & Claims
BIP-39 wallet generation, Keystore storage, Merkle proof claim flow.
78%
Why 78%
Web3j on Android is functional but has rough edges — documentation is sparse, and edge cases with gas estimation on L2s can be tricky. Android Keystore integration for key storage is well-documented but differs across OEMs. The claim flow itself (fetch proof, build tx, sign, submit) is straightforward once the infrastructure is working.
Conversion
Points → Token Migration
Snapshot point balances, generate Merkle tree, enable on-chain claims for early users.
92%
Why 92%
This is a database query (snapshot balances) + Merkle tree generation (proven algorithm) + one on-chain transaction (setMerkleRoot). Grass, Blur, and EigenLayer all did this successfully. The remaining 8% risk is UX friction in the claim flow causing user confusion or abandoned claims.
